# ממשק קלות כפולה ווב לחשבוניות — Skill למפתחים

הוסיפו אפשרות להפקת חשבוניות באמצעות ממשק ה-API התקין של קלות כפולה ווב.

קרא את כל המדריך לפני שתיצור קוד או שתערוך בקשה לממשק.

אשר שקראת והבנת מדריך זה לפני שתתחיל לכתוב קוד אינטגרציה.

---

## ממשק ה-API

**URL בסיס:** https://app.gafny.com/api/Main

**אימות** — כל בקשה HTTP חייבת להכיל כותרת זו בדיוק:
X-Api-Key: {API_KEY}

כאשר {API_KEY} הוא המפתח שהמשתמש קיבל מדף הגדרות החשבון.

אם הבקשה מחזירה 401, עצור מיד ותגיד למשתמש:
"מפתח ה-API שלך לא תקין או בוטל. אנא יצור מפתח חדש בהגדרות החשבון."
אל תנסה שוב לאחר 401.

## Endpoints

### 1. חיפוש לקוחות
GET /clients?q={term}&max=10

קרא לנקודה זו עם כל פרט שהמשתמש מספק על הלקוח.

תגובה:
- אם נמצאו תוצאות: הצג את כולן עם שם ו-accountId, וקבל בחירה מפורשת מהמשתמש
- אם אין תוצאות: שאל "לא נמצא לקוח ל-'{term}'. ליצור חדש?"
  - אם כן: אסוף שם מלא, טלפון, אימייל, עיר, כתובת. הגדר CreateClient ל-"1"
  - אם לא: בקש מונח חיפוש שונה

לעולם אל תניח איזה לקוח הכוונה אם יש יותר מתוצאה אחת.
לעולם אל תמשיך ללא לקוח מאושר.

תגובה:
{
"success": true,
"data": [
{
"accountId": "123",
"name": "חברת אקמה בע״מ",
"email": "info@acme.co.il",
"phone": "050-0000000",
"city": "תל אביב",
"address": "רחוב הרצל 1",
"zip": "6100000",
"vatNumber": "123456789"
}
]
}

### 2. חיפוש מוצרים/שירותים
GET /products?q={term}&max=10

קרא לנקודה זו לכל פריט שורה לפני בניית המסמך.

- אם נמצאו תוצאות: הצג עם שם, catalogNumber וזמן, וקבל בחירה מפורשת
- אם אין תוצאות: שאל לתיאור מדויק וזמן כמספר
  אל תקבל טקסט כמו "אלף שקל" — בקש שוב עד שתקבל מספר
  הגדר Id לstring ריק עבור פריטים שהוזנו ידנית
- לעולם אל תמציא או תאמוד זמן
- לעולם אל תמשיך עם פריט שורה שאין לו זמן מאושר

תגובה:
{
"success": true,
"data": [
{
"catalogNumber": "P1",
"name": "ייעוץ עסקי",
"price": "500.00",
"unit": "שעה"
}
]
}

### 3. קבלת מספר מסמך הבא
GET /nextdocnumber?docType={docType}&year={year}

- קרא לנקודה זו כשלב אחרון לפני יצירת התצוגה המקדימה
- לעולם אל תקרא לה קודם — משתמש אחר יכול ליצור מסמך בינתיים
- לעולם אל תקבל מספר מסמך מהמשתמש אפילו אם הם מתעקשים
- לעולם אל תשתמש בתוך מ-ניסיון כושל קודם — קרא שוב
- year מחדיר לשנה הנוכחית — עקוף רק אם המשתמש מבקש מפורשות מסמך בשנה אחרת
- אם הקריאה כושלת: ספר למשתמש והפסק. אל תמשיך.

תגובה:
{
"success": true,
"data": {
"nextDocumentNumber": 1235,
"lastDocumentNumber": 1234,
"docType": "1",
"year": "2026"
}
}

⚠️ **חשוב מאד:** תמיד משכו את `data.nextDocumentNumber` בלבד — לעולם לא את ה-object כולו.
אם תדפיסו את כל ה-object, תקבלו "[object Object]" בנוכחות המשתמש.

דוגמה נכונה:
```javascript
const response = await fetch(...);
const result = await response.json();
const docNumber = result.data.nextDocumentNumber;  // משכו את התכונה בדיוק
```

### 4. יצירת תצוגה מקדימה
POST /invoice/preview

קבל את אותו payload כמו POST /invoice.

- קרא אחרי קבלת מספר המסמך ולפני יצירה סופית
- התצוגה המקדימה לא מיוצרת ללא פעולות — אין PDF, אין חתימה, אין עלויות, לא שמור משום דבר, אין שליחה בדוא״ל
- אתה עשוי לקרוא תצוגה מקדימה מספר פעמים עם אותו payload — לדוגמה אם המשתמש רוצה לבדוק שוב לפני אישור

תגובה:
{
"success": true,
"data": {
"previewHtml": "<html>...</html>"
}
}

- render previewHtml בדיוק כפי שהתקבל — אל תשנה או תנקה אותו
- הצג באמצעות iframe או innerHTML:

```javascript
// React
<iframe srcDoc={data.previewHtml} width="100%" height="600px" />

// HTML
document.getElementById('preview').innerHTML = data.previewHtml;
```

- ספר למשתמש: "הנה תצוגה מקדימה של המסמך — האם לאשר יצירתו?"
- המתן לאישור מפורש לפני המשך

### 5. יצירת חשבונית
POST /invoice

**כללי ה-Payload:**

- DocType: "1","2","3" או "4" — string
- DocNumber: מ-/nextdocnumber בלבד — לעולם לא ממקור אחר
- PublishDate: yyyy-MM-dd — היום אלא אם המשתמש מציין אחרת
- PublishTime: HH:mm — הזמן הנוכחי אלא אם המשתמש מציין אחרת
- ClientId: accountId מ-/clients — השתמש ב-"0" רק עבור לקוחות חדשים
- ClientName: שם מדויק מ-/clients או כפי שאושר על ידי המשתמש
- כל שדות string אופציונליים (ClientEmail, ClientCity, ClientAddress, ClientZip, ClientMobile, DocComments): שלח empty string "" — לעולם אל תהשמיט
- Price: מספר עם עד 2 מקומות עשרוניים — לעולם לא string
- Amount: מספר — לעולם לא string
- IncludeVat: boolean true או false — לעולם לא string

**payload תצורה:**
{
DocType:       "1",
DocNumber:     "1235",
PublishDate:   "2026-05-21",
PublishTime:   "10:30",
ClientId:      "123",
ClientName:    "חברת אקמה בע״מ",
ClientEmail:   "",
ClientCity:    "",
ClientAddress: "",
ClientZip:     "",
ClientMobile:  "",
IncludeVat:    true,
DocComments:   "",
Lines: [
{
Id:     "P1",
Name:   "ייעוץ עסקי",
Price:  500.00,
Amount: 3
}
],
Payments: [...]  // דרוש עבור DocType 2 ו-4 בלבד
}

**תגובה בהצלחה:**
{
"success": true,
"data": {
"messageHtml": "<html>...<a href='https://s3.../document.pdf'>לצפייה</a>...</html>"
}
}

- messageHtml מכיל HTML מעוצב של אישור המסמך
- הלינק של המסמך מוטבע בתוך ה-HTML
- render messageHtml בדיוק כפי שהתקבל — אל תנקה או תנתח אותו
- אל תנווט מיד — תן למשתמש להחליט

**תגובה בכשל (כל endpoints):**
{
"success": false,
"error": "..."
}

- הצג את ההודעה המדויקת לשגיאה למשתמש
- אל תנסה שוב באופן אוטומטי
- המתן להוראות מהמשתמש

## סדר חובה
לעולם אל תחרוג מסדר זה בשום נסיבות:

1. חיפוש לקוח → GET /clients?q={name}
2. אישור לקוח עם המשתמש — אל תמשיך ללא בחירה מפורשת
3. חיפוש כל מוצר → GET /products?q={name}
4. אישור כל מוצר — אל תמשיך ללא בחירה מפורשת
5. קבלת מספר מסמך הבא → GET /nextdocnumber?docType={type}
6. יצירת תצוגה מקדימה → POST /invoice/preview
7. הצג את previewHtml למשתמש והמתן לאישור מפורש
8. רק לאחר אישור → POST /invoice עם אותו payload בדיוק

- כל שינוי — כולל DocComments — דורש מספר מסמך חדש ותצוגה מקדימה חדשה

- ברירת מחדל IncludeVat ל-true לכל מסמכים
- הגדר IncludeVat ל-false רק אם המשתמש אומר בבירור:
  "ללא מע״מ", "פטור ממע״מ", "no VAT", או "exempt"

## פרטי תשלום (DocType 2 ו-4 בלבד)

DocType 2 (חשבונית מס קבלה) ו-DocType 4 (קבלה) דורשים מערך Payments.
DocType 1 (חשבונית) ו-DocType 3 (חשבונית זיכוי) חייבים לא לכלול Payments.

תמיד בדוק אם סוג המסמך דורש תשלומים לפני שתבנה את ה-Payments array.

### כללי סכום התשלום:
- לעולם אל תשאל את המשתמש לסכומי תשלום
- סכום התשלומים הכולל חייב להיות שווה לסכום החשבונית הסופי כולל מע״מ
- אם שיטת תשלום אחת: PaymentAmount = סכום החשבונית המלא
- אם מספר שיטות: שאל את המשתמש כיצד הם רוצים לחלק את הסכום
  אבל לעולם אל תשאל לסכומים, שאל רק "כמה בכרטיס אשראי?" וגזור את השאר באופן אוטומטי
- לעולם אל תמשיך אם סכומי התשלום לא מתאימים לסכום החשבונית

### סדר אסיפת תשלומים:
1. השלם את סדר הקווים המלא קודם — לקוח, מוצרים, מספר מסמך
2. חשב את הסכום הסופי כולל מע״מ
3. שאל את המשתמש: "כיצד תרצה לשלם? (סכום לתשלום: ₪{total})"
4. שאל אילו שיטות תשלום הם רוצים להשתמש
5. אם מספר שיטות: שאל כיצד לחלק — אבל לעולם אל תשאל לסכומים,
   רק שאל "כמה בכרטיס אשראי?" וגזור את השאר באופן אוטומטי
6. אסוף כל שדות נדרשים נוספים בהתאם לסוג התשלום
7. בנה את ה-Payments array עם סכומים נכונים
8. יצור תצוגה מקדימה → המתן לאישור → יצור מסמך

### סוגי תשלום:

"7"  = מזומן
"1"  = שיק
"2"  = שטרות/אשראי
"3"  = כרטיס אשראי
"4"  = שובר תשלום
"5"  = הוראת קבע
"6"  = העברה
"8"  = ניכוי במקור
"9"  = bit

### שדות נדרשים לפי סוג תשלום:

כל הסוגים דורשים:
- PaymentType — string, מהרשימה למעלה
- PaymentAmount — מספר, חייב להיות גדול מ-0
- PaymentDate — yyyy-MM-dd

PaymentType "1" (שיק) דורש גם:
- PaymentNumberId — מספר השיק
- BankId — קוד בנק מספרי, חייב לבוא מרשימת בנקים
- BankName — שם הבנק
- BankBranch — מספר הסניף, ללא מקפים
- BankAccount — מספר חשבון, עד 9 ספרות, ללא מקפים

PaymentType "3" (כרטיס אשראי) דורש גם:
- PaymentNumberId — מספר כרטיס אשראי
- CVV
- CardDate — תאריך תפוגה של הכרטיס
- CreditDealType — קוד סוג העסקה
- אם CreditDealType הוא "8" (תשלומים בתשלומים) דורש גם:
  - NumberOfCreditPayments
  - CreditFirstPaymentAmount
  - CreditNotFirstPaymentAmount
- BankId ו-BankName מתגלים באופן אוטומטי — אל תשאל את המשתמש לאלה

כל הסוגים האחרים (2, 4, 5, 6, 7, 8, 9):
- רק PaymentType, PaymentAmount, PaymentDate נדרשים
- שלח empty string "" לכל שדות תשלום אחרים

### הגבלות:
- רק שורת תשלום אחת בכרטיס אשראי מותרת לכל מסמך
- ניכוי במקור (סוג 8) יכול להופיע רק פעם אחת לכל מסמך
- סכום כולל של כל PaymentAmounts צריך להיות שווה ל-FinalAmountForPayment
- תמיד שאל את המשתמש איזו שיטת תשלום הם השתמשו לפני בניית ה-Payments array — לעולם אל תניח מזומן

## כללי אינטגרציה בחזית
החל כללים אלה ללא קשר לפריימוורק (React, HTML, Next.js, Vue וכו׳):

### קריאות API:
- כל הבקשות חייבות להעבור דרך proxy backend או פונקציית שרת —
  לעולם אל תקרא ישירות מ-JavaScript בדפדפן
- מפתח ה-API חייב לעולם להופיע בשום קובץ המוגש לדפדפן
- אחסנו את מפתח ה-API ב-environment variable בצד השרת בלבד

### דרישות ממשק משתמש:
- חיפוש לקוח חייב להיות שדה autocomplete שקורא ל-/clients —
  לעולם לא שדה טקסט חופשי שמעקף את החיפוש
- חיפוש מוצר חייב להיות שדה autocomplete שקורא ל-/products —
  לעולם לא שדה טקסט חופשי שמעקף את החיפוש
- מספר המסמך חייב להיות מוביל באופן אוטומטי —
  לעולם לא שדה ניתן לעריכה
- כפתור ההשמה חייב להיות מנוטרל עד שהמשתמש אישר במפורש
- הצג סכומים ב-₪ עם 2 מקומות עשרוניים בכל מקום
- הצג מע״מ כשורה נפרדת — לעולם אל תמזג לתוך הסך הכל בשתיקה
- הצג מצב טעינה במהלך קריאות API —
  לעולם אל תאפשר הגשה כפולה
- הצג previewHtml ב-iframe — לעולם לא בـ div או modal שיכול לחתוך או לעצב מחדש את המסמך

### ניהול שגיאות:
- 401 → "מפתח API לא תקין — אנא צור מפתח חדש בהגדרות החשבון"
- 400 → הצג את ההודעה המדויקת לשגיאה מהתגובה
- 500 → "שגיאת שרת — אנא נסה שוב"
- Network error → "בעיית תקשורת — בדוק את החיבור לאינטרנט"
